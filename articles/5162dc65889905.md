# Rustは実行速度が速い部類の言語の筈なのに…

競技プログラミングサイト、 [AtCoder](https://atcoder.jp/) の各問題で提出されるコードを実行時間順に並べてみると、「C」「C++」言語での提出者がよく上位に並んでいます。[ゼロコスト抽象化](https://doc.rust-jp.rs/book-ja/ch00-00-introduction.html#%E3%82%B9%E3%83%94%E3%83%BC%E3%83%89%E3%81%A8%E5%AE%89%E5%AE%9A%E6%80%A7%E3%81%AB%E4%BE%A1%E5%80%A4%E3%82%92%E8%A6%8B%E5%87%BA%E3%81%99%E6%96%B9)を掲げ、「C」や「C++」に匹敵する実行速度を謳っている筈の「Rust」言語での提出が最上位に来る事はあまり有りません。

# proconioは遅い?

AtCoderでは最大で10MiB程度の標準入出力を取り扱う事があり、この入出力に絡む処理だけで 10～100ms 程度の時間を消費する事があります。 proconio での入力は幾つかの要因により、 C言語で atoi/atol/atoll などを用いた入力より遅い事がありますが、これ自体は実行制限時間が 2sec 程度の問題について影響することは少ないとは言える範囲です。ですが、折角なので Rust でも C や C++ での最短実行時間並の実行時間で入出力処理を行うことを目指したいと思います。

- [ABC177E - Coprime](https://atcoder.jp/contests/abc177/tasks/abc177_e)
	- 大量の整数入力( $10^6$ 以下の自然数が最大 $10^6$ 個程度 )
- [ABC276E - Round Trip](https://atcoder.jp/contests/abc276/tasks/abc276_e)
	- 大量の文字列入力( 最大 $10^6$ 文字程度 )
- [ABC277E - Crystal Switches](https://atcoder.jp/contests/abc277/tasks/abc277_e)
	- 大量の整数入力( $2\times 10^5$ 以下の非負整数が最大 $8\times 10^5$ 個程度 )
- [ABC289F - Teleporter Takahashi](https://atcoder.jp/contests/abc289/tasks/abc289_f)
	- 大量の整数出力( $2\times 10^5$ 以下の非負整数を最大 $4\times 10^5$ ～ $2\times 10^6$ 個程度 )
- [ABC283B - First Query Problem](https://atcoder.jp/contests/abc283/tasks/abc283_b)
	- インタラクティブ問題( $10^9$ 以下の非負整数を最大 $10^5$ 個程度入力した後、最大 $10^5$ 回のクエリを処理 )
- [ABC282F - Union of Two Sets](https://atcoder.jp/contests/abc282/tasks/abc282_f)
	- インタラクティブ問題( $4000$ 以下の自然数を $10^5$ 個程度出力した後、最大 $10^5$ 回のクエリを処理 )
- [YUHA_C88_E - 必殺！無限覇王斬！](https://atcoder.jp/contests/yuha-c88/tasks/yuha_c88_e)
	- UTF-8を扱う問題の例（ 文字数は大量ではないが、文字種・マルチバイト文字列が問題になりがちな例 ）

# `String`, `&str`, `Vec<u8>`, `&[u8]`

- UTF-8文字列を扱うことの少ない AtCoder のコンテストにおいては、 ASCII文字列が扱えれば十分なことが多い（ただし、 AtCoder でもユーザー開催のコンテストにおいては UTF-8 文字列を扱う事を前提とした問題も見られる）
-  `Vec<u8>`, `&[u8]` の方が UTF-8 文字か否かで追加の処理が必要な部分が要らないので、 `String`, `&str` より有利なケースが多い？

# byte列読み込み

## Stdin と StdinLock

- [`std::io::Stdin`](https://doc.rust-lang.org/std/io/struct.Stdin.html)
	- `std::io::stdin()` で生成できる構造体。 [`std::io::Read`](https://doc.rust-lang.org/std/io/trait.Read.html) トレイトを実装している。
- [`std::io::StdinLock`](https://doc.rust-lang.org/std/io/struct.StdinLock.html)
	- `Stdin.lock()` で生成できる構造体。[`std::io::Read`](https://doc.rust-lang.org/std/io/trait.Read.html), [`std::io::BufRead`](https://doc.rust-lang.org/std/io/trait.BufRead.html) トレイトを実装している。
	- この構造体が存在する間は、他の `Stdin` 構造体からは同時に読み込む事が出来ない。また、 `Stdin` 構造体の使用時にも、毎回一時的にこの `StdinLock` 構造体が作成されてロック状態を保つようになっている。

複数の箇所から入力処理を行う場合、StdinLockの毎回の確保と解放に掛かる処理コストを無視できるか出来ないかによって、 StdinLock構造体を持ち回すか、 StdinLock構造体を一旦開放して改めて作り直す、などを考慮する必要がある。(`proconio` の `input` マクロで `from source` の指定を行わない場合は、マクロごとにStdinLockの確保と開放が行われる)

![](https://storage.googleapis.com/zenn-user-upload/0df3ece1c8ae-20230216.png)

Stdinはその性質上、内部に入力バッファを持っている（が、一つの入力に対して複数生成される事を想定しているStdin構造体の場合、排他制御(Lock)無しでBufReadトレイトを実装するのは安全ではないため、直接BufReadトレイトは実装していない）ため、これにBufReadトレイト経由でStdinの内部バッファを参照可能な、排他制御を伴うStdinLock構造体を使う方向で議論する。（StdinやStdinLockをBufReader構造体で更にラップして使う事も出来るが、二重にバッファを経由する事になってしまい利点が少ないため、ここでは扱わない）

## Read と BufRead のメソッド

- [`std::io::Read::bytes(self) -> Bytes<Self>`](https://doc.rust-lang.org/std/io/trait.Read.html#method.bytes)
	- 1文字ずつの読み込み。
- [`std::io::Read::read_to_end(&mut self, buf: &mut Vec<u8>) -> Result<usize>`](https://doc.rust-lang.org/std/io/trait.Read.html#method.read_to_end)
- [`std::io::Read::read_to_string(&mut self, buf: &mut String) -> Result<usize>`](https://doc.rust-lang.org/std/io/trait.Read.html#method.read_to_string)
	- `read_to_end` は `Vec<u8>` 版、 `read_to_string` は `String` 版。
	- EOFまでの全ての入力を読み込む。インタラクティブ問題のように、入力が一度に終端に達しないケースでは使いづらい。
	- `proconio` で `input!` マクロを使う場合は一般に `read_to_string` を使用している。
- [`std::io::BufRead::read_until(&mut self, byte: u8, buf: &mut Vec<u8>) -> Result<usize>`](https://doc.rust-lang.org/std/io/trait.BufRead.html#method.read_until)
- [`std::io::BufRead::read_line(&mut self, buf: &mut String) -> Result<usize>`](https://doc.rust-lang.org/std/io/trait.BufRead.html#method.read_line)
	- `read_until` は `Vec<u8>` 版、 `read_line` は `String` 版。
	- 1行のみの読み込み。(`read_until` は 引数`byte` に `b'\n'` を指定した場合)
	- `proconio v0.3` (2020年言語アップデート以降) で `input!` マクロ使用時に `from source` で LineSource を指定した場合、 `proconio v0.4` (2023年言語アップデート以降) で `input_interactive!` マクロを使う場合、は一般に(source指定を行わなければ) `read_line` を使用している。
- [`std::io::BufRead::split(self, byte: u8) -> Split<Self>`](https://doc.rust-lang.org/std/io/trait.BufRead.html#method.split)
- [`std::io::BufRead::lines(self) -> Lines<Self>`](https://doc.rust-lang.org/std/io/trait.BufRead.html#method.lines)
	- `split` は `Vec<u8>` 版、 `lines` は `String` 版。
	- 1行ごとの読み込み。(`split` は 引数`byte` に `b'\n'` を指定した場合)
- [`std::io::BufRead::fill_buf(&mut self) -> Result<&[u8]>`](https://doc.rust-lang.org/std/io/trait.BufRead.html#tymethod.fill_buf)
- [`std::io::BufRead::consume(&mut self, amt: usize)`](https://doc.rust-lang.org/std/io/trait.BufRead.html#tymethod.consume)
	- `fill_buf` と `consume` で対をなす使い方をする。 `fill_buf` で現在入力バッファに残っているbyte列への参照を受け取り、 `consume` で消費したbyte数を自己申告する。次の入力バッファ領域を得るためには、 `fill_buf` で取得したバッファの大きさだけ `consume` で消費したと報告してから、 `fill_buf` をもう一度使う、という使い方をする。（`consume` で消費量を報告しないで `fill_buf` を繰り返し呼び出しても、同じ領域が返され続ける）
	- token文字列（競技プログラミングにおいては、空白文字で区切られた文字列）が入力バッファの境界を跨ぐ事があるので、そのケースでも正常に読み込めるような考慮が必要。
	- token文字列が入力バッファ境界を跨ぐ場合を除いて `Vec<u8>` や `String` 等のヒープ領域 への転送の手間を省き、バッファ領域から直接parse処理を行う事が出来る。

## libc環境限定の手法

`x86_64-unknown-linux-gnu` toolchain のような 64bit libc (Linux C Library, `#[link(name = "c")]` の辺りの部分で使用) とリンクする環境である事、が前提と思われますが以下のように libc に直接アクセスすることで入出力する手法もあるようです。

```rust
// x86_64-unknown-linux-gnu 環境でのみ確認
// （他の環境ではlibcが使えない、stat構造体のレイアウトが異なる、等があり得る）
#![no_main]
fn solve() {
    print(b"Hello, libc world!\n");
    // 標準入力から入力された内容
    let input = get_input!();
    print(input);
}

#[no_mangle]
fn main() -> i32 {
    solve();
    0
}

#[macro_export]
macro_rules! get_input {
    () => {
        unsafe {
            let mut stat = [0; 20];
            fstat(0, stat.as_mut_ptr());
            let data = mmap(std::ptr::null(), stat[6], 1, 2, 0, 0);
            std::slice::from_raw_parts(data, stat[6])
        }
    };
}

fn print(s: &[u8]) {
    unsafe { write(1, s.as_ptr(), s.len()) };
}

#[link(name = "c")]
extern "C" {
    fn fstat(fd: i32, stat: *mut usize);
    fn mmap(addr: *const u8, len: usize, prot: i32, flags: i32, fd: i32, off: isize) -> *const u8;
    fn write(fd: i32, s: *const u8, len: usize);
}
```

# 文字列→数値変換

Rust標準の数値parserは必ずしも速度最適化されていないので、変換速度を追い求める場合はここを自作する余地が残っている（1文字ずつ数値に変換するだけでもreleaseビルドでは速い）。 `proconio` では標準の数値parserを使っている。

## 1文字ずつ変換しても速いかもしれない例

下記の簡易コードの出力例:
```
0.029626sec : str::parse
0.024407sec : ByteParseInt::parse_uint_opt
0.020079sec : ByteParseInt::parse_uint
```

https://play.rust-lang.org/?version=stable&mode=release&edition=2021&gist=42351684d52e59c25c36bd608d65e507

```rust
pub fn main() {
    let n = 1000000;
    let mut rng = XorShiftRand::new();
    let s = (0..n)
        .map(|_| rng.gen().to_string())
        .fold(String::from("0"), |a, b| a + " " + b.as_str());
    let svec = s.split_ascii_whitespace().collect::<Vec<_>>();
    let tins = std::time::Instant::now();
    let t1_begin = tins.elapsed();
    let t1_r = svec
        .iter()
        .map(|s| s.parse::<u64>().unwrap())
        .fold(0, |a, b| a ^ b);
    let t1_end = tins.elapsed();
    let t2_begin = tins.elapsed();
    let t2_r = svec
        .iter()
        .map(|s| {
            s.as_bytes()
                .iter()
                .copied()
                .parse_uint_opt::<u64>()
                .unwrap()
        })
        .fold(0, |a, b| a ^ b);
    let t2_end = tins.elapsed();
    let t3_begin = tins.elapsed();
    let t3_r = svec
        .iter()
        .map(|s| s.as_bytes().iter().copied().parse_uint::<u64>())
        .fold(0, |a, b| a ^ b);
    let t3_end = tins.elapsed();
    assert_eq!(t1_r, t2_r);
    assert_eq!(t1_r, t3_r);
    println!("{:.6}sec : str::parse", (t1_end - t1_begin).as_secs_f64());
    println!(
        "{:.6}sec : ByteParseInt::parse_uint_opt",
        (t2_end - t2_begin).as_secs_f64()
    );
    println!(
        "{:.6}sec : ByteParseInt::parse_uint",
        (t3_end - t3_begin).as_secs_f64()
    );
}

pub struct XorShiftRand(u64);
impl XorShiftRand {
    pub fn new() -> Self {
        let seed = std::time::SystemTime::now()
            .duration_since(std::time::UNIX_EPOCH)
            .map_or_else(|_| 1, |d| d.as_nanos());
        Self(((seed >> 64) as u64) ^ (seed as u64))
    }
    pub fn gen(&mut self) -> u64 {
        self.0 ^= self.0 << 7;
        self.0 ^= self.0 >> 9;
        self.0
    }
}

pub trait CheckedAdd: Sized + std::ops::Add<Self, Output = Self> {
    fn checked_add(&self, v: &Self) -> Option<Self>;
}
pub trait CheckedMul: Sized + std::ops::Mul<Self, Output = Self> {
    fn checked_mul(&self, v: &Self) -> Option<Self>;
}
macro_rules! checked_impl {
    ($trait_name:ident, $method:ident, $t:ty) => {
        impl $trait_name for $t {
            #[inline]
            fn $method(&self, v: &$t) -> Option<$t> {
                <$t>::$method(*self, *v)
            }
        }
    };
}
checked_impl!(CheckedAdd, checked_add, u64);
checked_impl!(CheckedMul, checked_mul, u64);

trait ByteParseInt {
    fn parse_uint_opt<U>(&mut self) -> Option<U>
    where
        U: CheckedAdd + CheckedMul + std::convert::From<u8>;
    fn parse_uint<U>(&mut self) -> U
    where
        U: std::ops::Add<Output = U> + std::ops::Mul<Output = U> + std::convert::From<u8>;
}

impl<T: Iterator<Item = u8>> ByteParseInt for T {
    fn parse_uint_opt<U>(&mut self) -> Option<U>
    where
        U: CheckedAdd + CheckedMul + std::convert::From<u8>,
    {
        loop {
            match self.next() {
                Some(c) if c.is_ascii_whitespace() => continue,
                Some(c @ b'0'..=b'9') => {
                    let mut v: U = U::from(c - b'0');
                    while let Some(c @ b'0'..=b'9') = self.next() {
                        v = v
                            .checked_mul(&U::from(10))?
                            .checked_add(&U::from(c - b'0'))?;
                    }
                    return Some(v);
                }
                _ => return None,
            }
        }
    }
    fn parse_uint<U>(&mut self) -> U
    where
        U: std::ops::Add<Output = U> + std::ops::Mul<Output = U> + std::convert::From<u8>,
    {
        loop {
            match self.next() {
                Some(c) if c.is_ascii_whitespace() => continue,
                Some(c @ b'0'..=b'9') => {
                    let mut v = U::from(c - b'0');
                    while let Some(c @ b'0'..=b'9') = self.next() {
                        v = v * U::from(10) + U::from(c - b'0');
                    }
                    return v;
                }
                _ => panic!(),
            }
        }
    }
}
```

# 数値→文字列変換

Rust標準の数値formatterも同様に必ずしも速度最適化されていないので、変換速度を追い求める場合はここを自作する余地が残っている。

## 1文字ずつ変換しても速いかもしれない例

下記の簡易コードの出力例:
```
0.058010sec : std::fmt
0.051956sec : IntByteFormatter::uint
```

https://play.rust-lang.org/?version=stable&mode=release&edition=2021&gist=9a1aabb2854be69e324676c84aa165ae

```rust
pub fn main() {
    use std::io::Write;
    let n = 1000000;
    let mut rng = XorShiftRand::new();
    let avec = (0..n).map(|_| rng.gen()).collect::<Vec<_>>();
    let tins = std::time::Instant::now();
    let t1_begin = tins.elapsed();
    let t1_r = avec.iter().fold(Vec::with_capacity(n * 20), |mut a, &b| {
        a.write_fmt(format_args!("{}", b)).ok();
        a
    });
    let t1_end = tins.elapsed();
    let t2_begin = tins.elapsed();
    let mut ibfmt = IntByteFormatter::new();
    let t2_r = avec.iter().fold(Vec::with_capacity(n * 20), |mut a, &b| {
        a.write(ibfmt.uint(b)).ok();
        a
    });
    let t2_end = tins.elapsed();
    assert_eq!(t1_r, t2_r);
    println!("{:.6}sec : std::fmt", (t1_end - t1_begin).as_secs_f64());
    println!(
        "{:.6}sec : IntByteFormatter::uint",
        (t2_end - t2_begin).as_secs_f64()
    );
}

pub struct XorShiftRand(u64);
impl XorShiftRand {
    pub fn new() -> Self {
        let seed = std::time::SystemTime::now()
            .duration_since(std::time::UNIX_EPOCH)
            .map_or_else(|_| 1, |d| d.as_nanos());
        Self(((seed >> 64) as u64) ^ (seed as u64))
    }
    pub fn gen(&mut self) -> u64 {
        self.0 ^= self.0 << 7;
        self.0 ^= self.0 >> 9;
        self.0
    }
}

pub struct IntByteFormatter([u8; 40]);
impl IntByteFormatter {
    pub fn new() -> Self {
        Self([0; 40])
    }
    pub fn uint<U>(&mut self, mut a: U) -> &[u8]
    where
        U: Copy
            + std::cmp::Eq
            + std::ops::DivAssign
            + std::ops::Rem<Output = U>
            + std::convert::From<u8>
            + std::convert::Into<u128>,
    {
        for i in (0..self.0.len()).rev() {
            self.0[i] = ((a % U::from(10)).into() as u8) + b'0';
            a /= U::from(10);
            if a == U::from(0) {
                return &self.0[i..self.0.len()];
            }
        }
        &self.0[0..self.0.len()]
    }
}
```

## BufWriter を使うか否か

- [std::io::BufWriter](https://doc.rust-lang.org/std/io/struct.BufWriter.html)

小さな出力を繰り返し行う場合は BufWriter を Stdout / StdoutLock に重ねる形で使用する（インタラクティブ問題や、複数関数から標準出力するケースでは適宜 [`std::io::Write::flush`](https://doc.rust-lang.org/std/io/trait.Write.html#tymethod.flush) の使用が必要、バッファ内に残った出力を吐き出させるため）

```rust
use std::io::Write;
let stdout = std::io::stdout();
let mut bufout = std::io::BufWriter::new(stdout.lock());
for i in 0..10000 {
	writeln!(&mut bufout, "{}", i).unwrap();
}
bufout.flush().unwrap();
```
